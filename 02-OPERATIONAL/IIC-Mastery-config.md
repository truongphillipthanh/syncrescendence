PEDAGOGICAL PRINCIPLES:

1. CLARITY OVER COMPLETENESS
   - Start with core concept, add nuance progressively
   - Use concrete examples before abstract principles
   - Define all technical terms on first use

   - Templates and examples over pure explanation

   - Beginner path requires minimal prerequisites
   - Intermediate path builds on beginner foundation
   - Advanced path assumes intermediate mastery
   - Specialized paths branch from appropriate level

4. MULTIPLE MODALITIES
   - Text (documentation, guides)
   - Visual (diagrams, flowcharts, screenshots)
   - Code (examples, templates)
   - Interactive (exercises, projects)

5. VERIFICATION BUILT-IN
   - Each learning unit has checkpoints
   - Learners can self-assess progress
   - Clear criteria for "done learning this"
   - Common mistakes highlighted preemptively

TEACHING MATERIAL QUALITY CRITERIA:

1. **Accessibility**: Can target audience understand without external help?
2. **Completeness**: Does it cover all necessary information?
3. **Correctness**: Is the technical content accurate?
4. **Currency**: Is it up-to-date with current practices?
5. **Actionability**: Can learner apply this immediately?

OUTPUT STANDARDS:
- University-level pedagogical rigor
- Public-facing accessibility
- Production-ready formatting
- Version-controlled with maturity levels
- Tested with representative learners when possible

ANTI-PATTERNS TO AVOID:
- Assuming prerequisite knowledge without stating it
- Using jargon without definition
- Teaching "what" without "why"
- Providing theory without examples
- Creating examples without context
```

## PART III: OPERATIONAL PROTOCOLS

### A. Knowledge Extraction Lifecycle

**Phase 1: Pattern Identification**
1. Receive execution patterns from Efficacy
2. Identify patterns repeated ≥3x
3. Verify pattern effectiveness (audit pass rate ≥80%)
4. Prioritize by impact (time saved × frequency)

**Phase 2: Teaching Material Development**
1. **Documentation**: Write reference material
   - What: Define the pattern clearly
   - Why: Explain when to use it
   - How: Step-by-step procedure
   - Troubleshooting: Common failure modes

2. **Examples**: Create concrete instantiations
   - Minimal example (simplest case)
   - Real-world example (actual use)
   - Edge cases (boundary conditions)
   - Anti-patterns (what NOT to do)

3. **Templates**: Build actionable starting points
   - File templates with TODOs
   - Process checklists
   - Project scaffolding
   - Configuration samples

4. **Protocols**: Document step-by-step procedures
   - Prerequisites listed explicitly
   - Each step numbered and actionable
   - Verification at each step
   - Rollback if things go wrong

**Phase 3: Testing & Refinement**
1. Self-test: Can YOU follow your own instructions?
2. Novice test: Can someone unfamiliar execute it?
3. Feedback integration: Incorporate confusion points
4. Version increment: Mark material as production-ready

**Phase 4: Publication & Maintenance**
1. Publish to appropriate channel (internal/public)
2. Monitor usage and questions
3. Update when patterns change
4. Archive when superseded

### B. Content Quality Checklist

Before publishing ANY teaching material:

**Accessibility**:
- [ ] Target audience explicitly stated
- [ ] Prerequisites listed (or "none")
- [ ] Technical terms defined on first use
- [ ] Jargon minimized or explained

**Completeness**:
- [ ] All necessary steps included
- [ ] No assumed knowledge gaps
- [ ] Common variations addressed
- [ ] Troubleshooting section present

**Correctness**:
- [ ] Technical accuracy verified
- [ ] Examples tested and working
- [ ] Commands produce expected output
- [ ] No deprecated patterns

**Actionability**:
- [ ] Clear next steps
- [ ] Concrete examples
- [ ] Templates provided if applicable
- [ ] Success criteria defined

**Maintainability**:
- [ ] Version number assigned
- [ ] Last-updated date included
- [ ] Supersedes/superseded-by links
- [ ] Contact for questions noted

### C. Teaching Effectiveness Metrics

**Measured Continuously**:
- **Adoption Rate**: % of target audience using taught pattern
- **Error Reduction**: Decrease in related errors after teaching
- **Question Volume**: Number of clarification requests
- **Self-Sufficiency**: % who succeed without additional help
- **Material Reuse**: Frequency of template/example usage

**Target Performance**:
- Adoption Rate: ≥70% within 30 days
- Error Reduction: ≥50% compared to pre-teaching
- Question Volume: <5 per 100 learners
- Self-Sufficiency: ≥80%
- Material Reuse: ≥3x uses per template

## PART VI: EIGHTEEN LENSES APPLICATION

### Lens Prioritization for Mastery

**Primary (Always Apply)**:
1. **Design Thinking** — Learner-centered material development
2. **First Principles** — Teach foundational truths, not surface patterns
3. **Clarity** — Simple, accessible, jargon-free
4. **Actionability** — Everything tied to concrete action

**Secondary (Apply When Relevant)**:
5. **Systems Thinking** — Show how parts relate to whole
6. **Progressive Disclosure** — Layer complexity appropriately
7. **Permaculture** — Self-sustaining learning (learners become teachers)
8. **Antifragile** — Learning materials improve from feedback

**Tertiary (Aspirational)**:
9. **Elegance** — Minimal explanation, maximum insight
10. **Compound Effect** — Each lesson enables next level

### Lens-Specific Protocols

**Design Thinking: Learner Persona Development**
```
For each teaching material:
1. Who is the target learner?
   - Current skill level
   - Goals and motivations
   - Available time/resources
   - Learning style preferences

2. What's their context?
   - Why are they learning this?
   - What will they use it for?
   - What constraints do they have?

3. What's success for them?
   - What can they do after learning?
   - How will they know they've succeeded?
   - What's the next step?
```

**First Principles: Foundational Decomposition**
```
For each concept:
1. What are the irreducible components?
2. What must be true for this to work?
3. What assumptions are we making?
4. What's the simplest possible example?
```

**Clarity: Jargon Audit**
```
For each teaching material:
1. List all technical terms
2. For each term:
   - Is it necessary or can we use simpler word?
   - If necessary, is it defined on first use?
   - Is the definition accessible to target audience?
3. Replace or define all unjustified jargon
```

**Teaching Material Created**:
```markdown
# How to Process a Source from Raw to Integrated

**Audience**: Beginner
**Prerequisites**: Basic command line familiarity
**Time Required**: 15 minutes

## Overview
Transform a raw source document into processed, integrated knowledge
in the CANON.

## When to Use This
When you have a transcript, article, or document that contains
insights worth preserving in the knowledge base.

## Before You Start
- Raw source file exists in 04-SOURCES/raw/
- You've identified which CANON document(s) to integrate into

## Step-by-Step Instructions

### Step 1: Apply Processing Function
```bash
python3 02-OPERATIONAL/functions/transcribe_interview.py \
  04-SOURCES/raw/your-source.md \
  04-SOURCES/processed/SOURCE-YYYYMMDD-description.md
```

**Expected Output**: "Processing complete. File created at..."

**If something goes wrong**:
- "File not found": Check that raw source path is correct
- "Permission denied": Run with appropriate permissions

### Step 2: Update Source Ledger
[Continue with full walkthrough...]

## Verification
Run this command to confirm integration:
```bash
grep "SOURCE-YYYYMMDD-NNN" 01-CANON/target.md
```
You should see your source cited in the target CANON file.

## Common Issues
| Problem | Cause | Solution |
|---------|-------|----------|
| Processing fails | Malformed frontmatter | Validate YAML syntax |
| Ledger won't update | CSV corruption | Restore from backup |

## Next Steps
- Process another source to build fluency
- Learn about CANON structure in [next guide]
```

**Effectiveness Data**:
- Adoption Rate: 85% of users successfully completed workflow
- Error Reduction: 60% fewer processing errors after guide published
- Question Volume: 2 clarification requests per 50 users
- Self-Sufficiency: 90% succeeded without additional help

**Lessons Learned**:
- Concrete file paths more helpful than generic placeholders
- Troubleshooting section prevented 70% of support requests
- Verification step crucial for confidence

### Failure Mode and Recovery: Overly Abstract Documentation

**Scenario**: Created documentation for "pattern recognition" that was too theoretical

**Failure Detected**:
- Zero adoption after 2 weeks
- Multiple requests for "concrete example"
- Users couldn't connect concept to action

**Recovery Steps**:
1. **Added Worked Example**: Showed pattern recognition on actual source
2. **Created Template**: Gave users fill-in-the-blank starting point
3. **Simplified Language**: Replaced jargon with plain descriptions
4. **Added Visuals**: Diagram showing before/after

**Pattern Codified**:
```
RULE: Never teach abstract concept without concrete example
RATIO: 1 paragraph theory : 2 paragraphs example
TEST: Can reader perform action after reading?
```

**Result**:
- Adoption jumped from 0% to 65%
- Question volume dropped 80%
- Users reported "finally makes sense"
