# Rosetta / Intent / Backlog Triad v1

**Date**: 2026-03-02  
**Status**: constitutional repair design  
**Purpose**: reconstitute the triad that should govern semantic coherence, executive direction, and program execution in the next Syncrescendent shell

---

## 0. Why This Exists

The current stack already has:

- constitutional config compilation
- runtime reconciliation
- exocortex governance
- ontology projection
- first successful CLI-to-web relay

But the three highest-order guidance artifacts are not equally healthy:

- **Rosetta** is cited as constitutional pedigree but not cleanly surfaced as a live authority.
- **Intent Compass** is conceptually central but operationally obscured.
- **Backlog** exists and functions, but is not consistently bound upward to the other two.

This document repairs that triad.

---

## 1. The Triad

## 1.1 Rosetta

**Role**
- constitutional vocabulary
- semantic DNA
- category grammar
- original architectural and conceptual pedigree

**What it governs**
- ontology category coherence
- canonical definitions
- naming and classification boundaries
- promotion alignment

**Federal analogue**
- constitutional definitions
- jurisprudential vocabulary

## 1.2 Intent Compass

**Role**
- executive steering
- current aims
- active campaigns
- directional priorities
- deferred commitments with explicit status

**What it governs**
- what the system is currently optimizing for
- what should be deferred, parked, or pursued
- what program work is justified now

**Federal analogue**
- executive agenda
- cabinet-level strategic direction

## 1.3 Backlog

**Role**
- programmatic execution queue
- sequencing layer
- translation from goals into work

**What it governs**
- what is next
- what owns which execution lane
- what depends on what

**Federal analogue**
- administrative program
- implementation portfolio

---

## 2. Current Pathology

The present repo exhibits a triad asymmetry:

- backlog has a live visible office
- Rosetta and Intent are mostly living as references, echoes, and pedigree rather than explicit live institutions

This creates several failure modes:

1. ontology rules refer to living concepts with unclear living files
2. backlog becomes queue-centric rather than constitutionally and teleologically bound
3. semantics drift because Rosetta authority is cited more than executed
4. new implementation work can proceed without clear intention lineage

---

## 3. The Repair Principle

The fix is not to recover a single old file path and declare victory.

The fix is to separate:

- **pedigree**
- **live authority**
- **archive**

Each of the three needs a living present-tense form and a preserved historical lineage.

---

## 4. Rosetta Reconstitution

## 4.1 Rosetta has two lawful forms

### Rosetta Pedigree

**Purpose**
- preserve the original conceptual seedbed and idiosyncratic originality

**Contains**
- historical formulations
- original conceptual inventions
- the semantic lineage of Syncrescendent terms

**Class**
- constitutional provenance

### Rosetta Live

**Purpose**
- the currently binding vocabulary and category law

**Contains**
- normalized live terms
- allowed meanings
- current semantic boundaries
- category relations needed by canon, ontology, and playbooks

**Class**
- constitutional source

## 4.2 Rosetta rules

1. Rosetta Live must stay compact and executable.
2. Rosetta Pedigree may be much larger and richer.
3. Rosetta Live should cite Rosetta Pedigree, not absorb all of it.
4. Ontology, canon, and backlog should bind to Rosetta Live; pedigree remains interpretive reserve.

## 4.3 Rosetta edge cases

- if a term exists in pedigree but not in live Rosetta, it is not currently binding
- if live Rosetta changes a term, the pedigree lineage remains intact
- if multiple historical variants exist, live Rosetta chooses the current authority but preserves aliases

---

## 5. Intent Compass Reconstitution

## 5.1 Intent Compass has two lawful forms

### Intent Pedigree

**Purpose**
- preserve historical directional lines, prior campaigns, and deferred strategic arcs

### Intent Live

**Purpose**
- the binding executive priority surface for the current shell

**Contains**
- active intentions
- current campaigns
- explicit statuses:
  - active
  - deferred
  - parked
  - superseded
  - completed

## 5.2 Intent rules

1. Backlog items must bind to Intent Live, not just free-float.
2. Deferred commitments must still remain queryable and visible.
3. Intent Live should be compact enough to steer work, not so large it becomes another archive.

## 5.3 Intent edge cases

- old intentions may remain in pedigree even when no longer active
- backlog items can bind to multiple intentions when truly cross-cutting, but should name a primary intention
- runtime or ontology artifacts do not originate intention; they only reflect it

---

## 6. Backlog Rebinding

## 6.1 Backlog as the triad executor

Every substantial backlog item should bind upward to:

- one or more Rosetta concepts
- one or more Intent identifiers
- an execution surface or playbook when applicable

## 6.2 Minimum fields for mature backlog items

- `id`
- `title`
- `status`
- `intent_ref`
- `rosetta_ref`
- `playbook_ref` when applicable
- `execution_surface`
- `lineage_or_origin`

## 6.3 Why this matters

Without this rebinding:

- backlog becomes generic task management
- ontology becomes semantically underconstrained
- harness playbooks drift from strategic purpose

---

## 7. Alignment with the Multiagent -> Ontology Pipeline

The triad should sit above and inside the working pipeline:

`Rosetta -> Intent -> Backlog -> Playbook/Execution -> Runtime/Event -> Ontology`

### Rosetta contribution
- semantic coherence
- category selection
- vocabulary boundaries

### Intent contribution
- teleological coherence
- strategic prioritization

### Backlog contribution
- executable sequencing
- ownership and surface assignment

### Ontology contribution
- typed projection of work and state already normalized through the above layers

Ontology is downstream from the triad. It must not become a substitute for any part of it.

---

## 8. Alignment with Harness and Context-Engineering Research

The harness research implies:

- context should be progressively disclosed
- instructions should be role-specific
- system prompts are compiled artifacts, not prose dumps
- prompt portability is a myth

That means the triad should not be injected wholesale into every harness.

Instead:

- Rosetta informs constitutional and semantic constraints
- Intent informs active priorities
- Backlog informs immediate work
- playbooks translate that into native harness grain

The triad is therefore upstream of harnesses, not a direct replacement for playbooks.

The next shell is not constitutionally healthy until:

- Rosetta exists both as pedigree and as live constitutional vocabulary
- Intent Compass exists both as pedigree and as live executive steering
- Backlog binds upward to both rather than operating as a free-floating queue

Until then, execution may work, but institutional coherence remains partial.
