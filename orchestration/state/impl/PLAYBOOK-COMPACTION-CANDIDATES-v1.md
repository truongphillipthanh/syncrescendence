# Playbook Compaction Candidates v1

**Date**: 2026-03-02  
**Status**: tranche-01 derivative  
**Purpose**: identify the first playbooks that should be compacted from current doctrine, lineage, runtime evidence, and neocorpus research

---

## 0. Why This Exists

The shell now has playbook law, but not yet finished playbooks.

The question is not "what could become a playbook eventually?"

The question is:

**which playbooks should be institutionalized first because they are both high-leverage and already sufficiently evidenced?**

---

## 1. Selection Criteria

A surface becomes a first-wave playbook candidate when:

1. it is already operationally important
2. it has enough repeated usage or doctrine to compact
3. failure on that surface causes meaningful shell drift or friction
4. the native grain is real enough that a universal doctrine would be inferior

---

## 2. First-Wave Candidates

## 2.1 Claude Code

**Why first-wave**
- strongest empirical and research base in the repo
- central constitutional role
- native context, skills, handoff, and configuration logic already extensively mined

**Source base**
- Claude Code neocorpus entries
- current config architecture
- handoff protocol
- context compaction doctrine
- progressive disclosure doctrine

**Playbook scope**
- how Claude Code should be configured
- what belongs in constitutional surfaces vs skills vs handoffs
- context-window management
- when to use sub-agents vs direct execution
- how to stage research and implementation work

## 2.2 OpenClaw / Ajna / Psyche

**Why first-wave**
- live runtime surface
- bridge between subscription cognition, channels, browser, and repo truth
- strong local empirical evidence now exists

**Source base**
- OpenClaw runtime repairs
- Ajna/Psyche identity split
- browser relay and channel integration work
- OpenClaw memory architecture research

**Playbook scope**
- auth modes
- runtime sync
- workspace deployment
- browser role
- event emission and reconciliation
- differences between Ajna and Psyche surfaces

## 2.3 Oracle / Grok

**Why first-wave**
- one of the oldest and most important external cognition surfaces
- prompting doctrine is already sophisticated and distinctive
- heavily used in ascertescence and architectural sensing

**Source base**
- historical Oracle prompt line
- current packet system
- content-proof / ugly-quote doctrine
- triangulation role design

**Playbook scope**
- how to ground Oracle
- when to use Oracle vs Augur/Perplexity
- what kinds of questions belong there
- how outputs should be assessed and reintegrated

## 2.4 Perplexity / Augur

**Why first-wave**
- currently one of the clearest verifier surfaces
- now already packetized and bridged
- central to triangulation and web-gap closure

**Playbook scope**
- verification role
- current-web capability checks
- citation handling
- limits vs Oracle
- when Perplexity is the right surface

## 2.5 Cowork / Claude in Chrome

**Why first-wave**
- first successful real CLI-to-web relay
- clearly distinct native grain from terminal Claude Code
- likely major execution surface for subscription-worker fabric v1

**Source base**
- CLI-WEB-GAP package
- cowork-v1 relay
- Hazel-backed finalization
- browser relay assessments

**Playbook scope**

- gesture-vs-script split
- when Cowork should execute vs when scripts should stage/finalize
- limits of Chrome-dependent execution

## 3. Second-Wave Candidates

These are important, but should likely follow the first wave:

## 3.1 Codex

Why second wave:
- important and live
- but current redesign work already relies on a functioning native usage pattern here, so the most urgent gap is less severe than for Cowork/OpenClaw/Oracle

## 3.2 Manus

Why second wave:
- live and useful
- but role is narrower and more bounded
- first wave should formalize the more foundational harness and cognition surfaces first

## 3.3 Obsidian

Why second wave:
- important cockpit/interface role
- but still secondary to the main coding, exocortex, and browser relay surfaces

## 3.4 Hazel

Why second wave:
- useful edge trigger surface
- but should remain subordinate to stronger operator law

---

## 4. Anti-Candidates for Early Playbook Work

Do not prioritize first-wave playbooks for:

- surfaces that are still mostly conceptual
- tools with little or no repeated operational evidence
- tools whose role is mostly covered by broader exocortex policy today

Examples:

- Feedcraft as a whole
- IIC surface family as a whole
- future headless worker fabric as a whole

These need stage-0 and architectural docs first, not full playbooks yet.

---

## 5. Suggested Compaction Order

1. Claude Code
2. OpenClaw
3. Oracle
4. Perplexity
5. Cowork
6. Codex
7. Manus
8. Obsidian
9. Hazel

This order reflects:

- importance,
- evidence maturity,
- and how foundational the playbook is for the rest of the shell.

---

## 6. Net Rule

The first playbooks should not be chosen by personal preference or tool novelty.

They should be chosen by:

- operational centrality,
- accumulated doctrine,
- repeated pain,
- and compaction readiness.
