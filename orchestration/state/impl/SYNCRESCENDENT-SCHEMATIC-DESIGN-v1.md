## 0. Why This Exists

This is not a cleanup memo.

This is a redesign blueprint for the shell that contains:

- the repo
- the machines
- the agent harnesses
- the scripts and automations
- the exocortex
- the browser relay layer
- the memory system
- the ontology layer
- the intake and compaction pipeline

The old shell accreted under real work. It proved many things and then crufted.

CRUSH / nucleosynthesis solved one part of that problem:

- flattening and reclassifying knowledge
- extracting a traversable neocorpus
- restoring semantic legibility to the knowledge substrate

But the architecture is larger than the corpus.

The same redesign logic now has to be applied to:

- constitutional law
- executive priorities
- program/backlog
- harness playbooks
- communications lineage
- scripts and automations
- runtime/exocortex/ontology boundaries

This document is the first-pass schematic for that reconstitution.

## 2. Governing Thesis

The correct redesign principle is:

**build the ideal Syncrescendent shell from scratch, then repopulate it losslessly from the old shell, compacting and promoting as needed, without destroying provenance.**

This implies:

1. original files remain as provenance unless confirmed duplicates
2. the new shell is designed by artifact law, not by convenience
3. compaction is built into the architecture rather than treated as cleanup
4. harnesses are worked with, not flattened into one fake universal agent
5. the system remains antifragile under capability shifts and paradigm changes

The shell must support molting.

## 3. The Highest-Order Shape

The ideal Syncrescendent shell has ten primary layers.

### 1. Metaphysics / Teleology

What cannot be traded away:

- worldview
- civilizational orientation
- sovereignty principles
- telos
- success criteria beyond immediate operations

This is the deepest invariant substrate.

### 2. Constitution

Shared law and definitions:

- constitutional axioms
- Rosetta / canonical vocabulary
- boundary contract
- ontology gate
- cross-surface routing law
- source-vs-generated law

This is not a compendium.
It is a compact legal layer.

### 3. Executive

Active direction and steering:

- Intent Compass
- live priorities
- current campaigns
- active directional vectors
- deferred commitments with explicit status

This is agenda, not law.

### 4. Program

Administrative and implementation layer:

- backlog
- implementation map
- delivery tranches
- sequencing
- ownership
- venues

This is where intentions become executable work.

### 5. Playbooks

Harness- and surface-specific doctrine:

- Claude Code
- Codex
- Gemini
- OpenClaw
- Oracle
- Perplexity
- Cowork
- Manus
- Obsidian
- future browser adapters, Feedcraft, IIC surfaces

Playbooks are where native grain is preserved.

### 6. Communications

Structured lineage artifacts:

- prompts
- responses
- dispatches
- handoffs
- retrospectives
- assessments
- derived outputs

This is the part the repo has under-formalized the most.

### 7. Execution

What actually does work:

- local scripts
- hooks
- skills
- workers
- agents
- browser relays
- tmux fabric
- automations
- launchd services

Execution is not authority.

### 8. Exocortex

Externalized cognitive and operational surfaces:

- SaaS tools
- browser-native surfaces
- APIs
- feed surfaces
- identity/account surfaces
- notebook and research surfaces

These are governed ingress/egress surfaces.

### 9. Registry / Projection

Typed mirrors and query surfaces:

- ontology
- dashboards
- API/domain
- reports derived from reconciled truth

This layer projects.
It does not legislate.

### 10. Archive / Provenance

Cold storage and superseded shells:

- corpus
- historical artifacts
- old shells
- superseded docs
- original files retained for archaeology

This remains lossless.

## 5. Rosetta Stone As Pedigree

Rosetta should not be treated as "just another missing file."

It is pedigree.

Its role is not merely to define terms, but to:

- carry the original idiosyncratic conceptual moves of the system
- ground ontology alignment
- preserve the project’s internal vocabulary from drift and flattening
- provide the semantic DNA from which canon, ontology, playbooks, and implementation all derive

The current repo symptom is severe:

- Rosetta is still cited as constitutional authority
- its live authoritative file path is unresolved

Therefore the redesign must distinguish between:

### Rosetta as pedigree

The historical and conceptual lineage:

- all prior resolved terms
- term evolution
- supersession chains
- crosslinks into canon and neocorpus

### Rosetta as constitutional surface

A live, compact, current vocabulary authority:

- canonical terms
- definitions
- category bindings
- ontology-relevant primitives
- references to deeper pedigree

The mistake would be either:

- collapsing Rosetta into a monolithic dead encyclopedia
- or reducing it to a tiny glossary that loses the project’s original conceptual inventions

The right design is:

- **pedigree store** for lineage
- **constitutional Rosetta surface** for active law

The same principle likely applies to Intent Compass.

## 6. Intent Compass And Backlog

### Intent Compass

The executive layer should hold:

- active intentions
- deferred intentions
- parked intentions
- resolved intentions
- campaign groupings
- priority and recency
- relation to constitutional vocabulary

Intentions are not tasks.
They are steering vectors.

### Backlog

The program layer should contain only executable work.

Every backlog item should bind upward to:

- one or more intention IDs
- relevant Rosetta / constitutional terms
- owning playbook or harness surface when applicable

Backlog items that float free of intention and vocabulary become mere queue clutter.

### Deferred commitments

The historical evidence is brutal:

- deferred commitments evaporate across sessions
- context compaction destroys navigability
- memory alone is insufficient

So the redesign must treat deferred commitments as:

- executive artifacts if they change steering
- program artifacts if they are implementation obligations

Never as mere memory notes.

## 7. Harness Thesis

The repo should explicitly embrace:

- prompt portability is a myth
- harness is the product
- progressive disclosure is mandatory
- skills, hooks, and layered configs are first-class

This means:

### Shared constitution must stay thin

The constitution should contain only what all harnesses must inherit.

### Harness playbooks must carry native grain

Each playbook should encode:

- cognitive function
- harness-specific defaults and anti-patterns
- operating formulas
- known failure modes
- valid invocation patterns
- what should be scripted versus improvised
- what can be entrusted to hooks, skills, or automation

### Generated surfaces remain generated

The existing config lesson remains binding:

- source is authority
- generated outputs are render artifacts
- never edit generated outputs directly

This principle should generalize beyond config:

- any rendered derivative must point back to source
- any compaction artifact must declare what it compacted from

## 8. Artifact Law

The redesign requires a strict legal distinction between artifact classes.

### A. Constitutional documents

Purpose:

- define law, vocabulary, gates, and invariants

Must never contain:

- runtime logs
- prompts
- responses
- session narrative

### B. Executive documents

Purpose:

- record current direction and intention

Must never contain:

- backlog details masquerading as strategy
- logs
- raw responses

### C. Program documents

Purpose:

- define executable work

Must never contain:

- ambiguous strategy without intention binding
- unresolved constitutional doctrine

### D. Playbooks

Purpose:

- encode harness- or surface-specific doctrine

Must never contain:

- live runtime state
- raw logs
- raw responses
- queue state

### E. Prompts / dispatches

Purpose:

- outbound communication packets

Must never contain:

- retrospective narrative
- canonical law
- unbounded hidden state

### F. Responses

Purpose:

- inbound external return artifacts

Must never be treated as:

- direct canon
- direct memory
- direct truth without assessment or reconciliation

### G. Handoffs

Purpose:

- state transfer across sessions

Must contain:

- what happened
- what remains
- what the next session must know
- key decisions and rationale

Must never become:

- permanent narrative memory

### H. Logs / retrospectives / kaizen

Purpose:

- extract lessons and quality signals

Must never become:

- narrative graveyards
- unbounded chronology

They should compact into:

- regulations
- anti-patterns
- automation
- quality management

### I. Outputs / assessments / scaffolds

Purpose:

- hold the product of work

Must declare:

- lineage
- source inputs
- whether they are transient, operational, or promotable

### J. Runtime artifacts

Purpose:

- support execution

Must not be allowed to become hidden authority.

## 9. Compaction Architecture

Compaction should be a built-in life cycle, not a janitorial afterthought.

### Logs compact into:

- quality protocols
- regulations
- anti-pattern guidance
- automation/hook opportunities
- maintenance rhythms

### Prompts and responses compact into:

- prompt primitives
- prompt engineering patterns
- playbook doctrine
- skills
- hooks
- reusable relay packet templates

### Research and source material compact into:

- neocorpus
- canon
- implementation deltas

### Runtime experience compacts into:

- operating laws
- scripts
- validators
- route rules

### Failed patterns compact into:

- cautionary exempla
- hard prohibitions
- test cases

### Key design rule

Compaction is not deletion.

It is:

- extraction
- compression
- promotion
- supersession
- retention of provenance

The shell should therefore support:

- raw provenance
- compacted doctrine
- supersession links
- active surface

all simultaneously.

## 10. Scripts As First-Class Citizens

Scripts are not incidental.

They are the execution operators of the shell.

The redesign should explicitly classify scripts into operator classes.

### 1. Stage

Scripts that prepare work:

- stage packets
- stage jobs
- stage relay artifacts

### 2. Validate

Scripts that check law and integrity:

- path checks
- artifact-class checks
- lineage checks
- generated/source drift checks

### 3. Reconcile

Scripts that move runtime truth back into repo truth:

- event reconciliation
- runtime snapshotting
- channel/runtime sync

### 4. Promote

Scripts that move artifacts upward:

- compaction
- synthesis
- promotion to neocorpus/canon
- ontology projection

### 5. Finalize

Scripts that close loops:

- relay finalizers
- commit gates
- post-run cleanup

### 6. Observe

Scripts that report health:

- local surface status
- machine status
- resolver/edge status
- agent runtime identity

### 7. Enforce

Hooks, CI, validators, and quarantine tools.

The shell should make these script classes explicit.

It should also define:

- who may call them
- which layers they may mutate
- what their input/output contracts are

Random standalone scripts are a pre-shell condition.

## 11. Intake And Insight Integration

Incoming insight is one of the least formalized but most important flows.

It should follow a single lifecycle:

### 1. Arrive

Surface:

- repo-local file
- web response artifact
- exocortex event
- note
- transcript
- research source

### 2. Classify

What kind of thing is it?

- provenance material
- dispatch/response
- output
- runtime state
- doctrine candidate
- constitutional candidate
- program candidate

### 3. Route

Send to the correct law-governed lane.

### 4. Compact / extract

If it is knowledge-bearing:

- corpus
- neocorpus
- canon candidate
- implementation delta

If it is operational:

- handoff
- retrospective
- quality protocol
- automation candidate

### 5. Bind upward

Every promoted insight should declare:

- which Rosetta terms it touches
- which intention(s) it serves
- which program or playbook it affects

### 6. Reconcile and project

Only then should it touch ontology or runtime-facing mirrors.

This should be the general intake law, not just a corpus process.

## 13. Edge Cases And Pain Points

The shell must explicitly handle the cases that previously hurt the system.

### A. Phantom paths

Historical pain:

- silent failure across many sessions

Design response:

- every constitutional/program/playbook path reference must validate mechanically

### B. Missing authority files

Historical pain:

- Rosetta and Intent Compass still cited while not cleanly resolving

Design response:

- authoritative surfaces must be mechanically discoverable
- pedigree and live authority must both exist

### C. Monolithic constitution

Historical pain:

- seared lessons accumulate until the constitution becomes unreadable

Design response:

- thin constitution
- deeper playbooks and doctrine layers
- progressive disclosure

### D. Lineage-as-memory

Historical pain:

- old session narrative clogs current operation

Design response:

- handoffs as transfer only
- retros compact to patterns
- no unbounded narrative memory

### E. Deferred commitment evaporation

Historical pain:

- promises decay between sessions

Design response:

- bind to executive/program layers
- explicit statuses and owners
- not just memory notes

### F. Race conditions in dispatch/ingest

Historical pain:

- watch-based and ad hoc loops caused duplicate/race failures

Design response:

- explicit queues
- explicit lifecycle states
- locking where needed
- one operator class per stage

### G. Stale documentation

Historical pain:

- docs describing worlds the repo no longer inhabits

Design response:

- freshness metadata
- validation against filesystem reality
- compaction of stale docs into historical archive if superseded

### H. Hidden authority surfaces

Historical pain:

- browser tabs, runtime caches, dotfiles, and external tools holding truth

Design response:

- boundary contract enforced
- runtime reconciles upward
- no second authority

### I. Browser relay fragility

Historical pain:

- gesture-heavy runs
- human anxiety about interference

Design response:

- clear gesture-vs-script split
- dedicated execution contexts
- mailbox/job queue law
- compaction of successful runs into operator playbooks/skills

### J. Script sprawl

Historical pain:

- powerful scripts with unclear authority and invocation law

Design response:

- operator classes
- designated locations
- explicit I/O contracts
- hooks/CI ownership

### K. Capability shifts

Historical pain:

- the environment changes faster than the shell

Design response:

- shell designed for molting
- compact doctrine, thin law, local playbooks
- scripts and relays treated as replaceable execution surfaces

## 15. Physical Instantiation Strategy

The physical migration should not begin with renaming everything.

It should begin with:

### Phase 1: constitutional recovery

- restore live Rosetta surface
- restore live Intent Compass surface
- define artifact classes
- define lineage metadata

### Phase 2: playbook layer

- extract per-harness doctrine from constitution, prompts, and proven patterns
- create state/local doctrine surfaces

### Phase 3: communications law

- define where prompts, responses, handoffs, outputs, and retros live
- quarantine deprecated and mixed lanes

### Phase 4: script/operator law

- classify scripts
- define ownership and invocation
- attach validators/hooks

### Phase 5: intake + compaction engine

- unify incoming insight routing
- define compaction/promotion flow beyond corpus only

### Phase 6: execution fabric hardening

- workers
- tmux
- browser adapters
- mini sandboxing

The rule is:

**logical shell first, migration second, rename last.**

## 16. Enforcement Philosophy

The shell must enforce itself.

That means:

- validator gates
- CI failures
- quarantine lanes
- anti-drift checks
- source-vs-generated checks
- lineage requirements

Soft norms are not enough.

The system’s history shows that if a deprecated lane is merely frowned upon, it will continue being used.

The shell must therefore be able to say:

- this artifact class is illegal here
- this file is authoritative but stale
- this output is missing lineage
- this generated file was edited directly
- this path is ghosted
- this prompt/response pair is orphaned

and stop the move until fixed.

## 17. Design Maxims

1. **Repo sovereignty without playbook law becomes brittle.**
2. **Playbooks without constitutional law become fragmentation.**
3. **Backlog without intention becomes bureaucratic drift.**
4. **Ontology without constitutional vocabulary becomes false precision.**
5. **Logs without compaction become lineage-as-memory.**
6. **Prompts and responses without lineage become tab-state.**
7. **Scripts without operator law become shadow infrastructure.**
8. **Runtime without reconciliation becomes a second reality.**
9. **Compaction without provenance becomes amnesia.**
10. **A shell that cannot molt will be broken by the next capability jump.**
