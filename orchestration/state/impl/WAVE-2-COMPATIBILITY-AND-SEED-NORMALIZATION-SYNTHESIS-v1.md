## 3. Adjudicated Collisions

### 3.1 Lane 03 overrides Lane 04 on seed and custody joins

Lane 04 drafted useful custody shapes, but it was operating against stale assumptions.

Lane 03 wins on:

- candidate ids
- merge family ids
- repo-relative source paths
- disposition targets
- `record_state`

No manifest, receipt, or registry artifact should carry Lane 04's earlier speculative promotion targets.

### 3.2 Root and communications edits must merge, not compete

Wave 2 created overlap across:

- Lane 01
- Lane 02
- Lane 07

That overlap is constructive, not contradictory.

The next wave should merge:

- Sigma compatibility wording
- dispatch-lane wording
- macro anchor wording

into one coherent front-door cleanup rather than stacking separate patch sets.

### 3.3 Live registry population remains out of scope for the next pass

Even with normalized rows in hand, the live registry and ledger should remain untouched until:

- custody artifacts are real and joinable
- any exact hash serialization ambiguity is ratified
- the next execution wave confirms that file paths and receipt paths are not hypothetical

### 3.4 Sigma physical sync is lawful only as exact-copy compatibility work

The first Sigma sync is no longer blocked in principle, but it is blocked against scope creep.

It must remain:

- bounded
- exact-copy by subtree
- source-preserving
- receipt-backed

If any file requires semantic editing to "fit" Sigma, that content belongs in a later compaction wave, not the sync wave.

### 3.5 Validator rollout remains downstream

Validators should not become mandatory before:

- compatibility docs are corrected
- custody artifacts exist
- at least one later wave inserts real registry rows and matching ledger events

- which public surfaces still lie

Wave 2 strengthens that doctrine by making the first real migration actions smaller and more truthful.
