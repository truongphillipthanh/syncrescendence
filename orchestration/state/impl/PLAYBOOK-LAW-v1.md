# Playbook Law v1

**Date**: 2026-03-02  
**Status**: constitutional redesign package  
**Purpose**: define what a playbook is, how playbooks differ from constitutional law and communications lineage, and how harness-native doctrine should be formalized in the next Syncrescendent shell

---

## 0. Why This Exists

One of the strongest validated theses in this repo is:

- the harness is the product
- prompt portability is a myth
- every serious agent surface has a native grain

That means the system should not try to flatten Claude Code, Codex, Gemini, OpenClaw, Oracle, Perplexity, Cowork, and Manus into one fake universal agent.

At the same time, Syncrescendence still needs:

- one constitutional source
- one truth plane
- one coherent governance structure

Playbooks are the layer that resolves this tension.

---

## 1. Definition

A playbook is a durable, surface-specific doctrine artifact that explains **how to work with a particular harness, agent surface, or execution environment while remaining inside Syncrescendent law**.

A playbook is not:

- the constitution
- a prompt
- a response
- a handoff
- a runtime snapshot
- a generic report

It is a **stable operating doctrine** for a specific native surface.

---

## 2. Why Playbooks Exist

The constitution should stay compact, shared, and cross-surface.

If every harness-specific nuance gets pushed into constitutional law, the constitution becomes:

- bloated
- increasingly contradictory
- less progressively loadable
- harder to compile
- harder to validate

Playbooks let the system preserve native grain without fragmenting sovereignty.

---

## 3. The Legal Role of a Playbook

Each playbook sits:

- below the constitution
- below executive intention
- below program priorities
- above prompts and runtime behavior

It is the "state law" layer in the federal architecture.

It translates:

- constitutional rules
- active intentions
- program requirements

into:

- harness-native practice
- context-loading patterns
- tooling usage patterns
- memory and handoff practice
- safe delegation behavior

---

## 4. What a Playbook Must Contain

Every mature playbook should eventually answer:

1. **What this surface is for**
2. **What it is bad at**
3. **How it manages context**
4. **How it should be loaded**
5. **What inputs it expects**
6. **What outputs it should produce**
7. **How it should hand off or return state**
8. **What operators/scripts/hooks it should use**
9. **What recurring anti-patterns must be blocked**
10. **What gets compacted from repeated usage into the playbook**

## 6. Playbooks vs Other Artifact Classes

## 6.1 Playbook vs Constitution

**Constitution**
- shared law
- global rules
- cross-surface invariants

**Playbook**
- local doctrine
- harness-native grain
- procedural specifics for a given surface

## 6.2 Playbook vs Prompt

**Prompt**
- single dispatch artifact
- concrete run instance

**Playbook**
- recurring doctrine
- reusable operating pattern

## 6.3 Playbook vs Log

**Log**
- records what happened

**Playbook**
- says how the surface should be used next time

## 6.4 Playbook vs Skill / Hook / Operator

**Skill / hook / operator**
- executable or quasi-executable mechanism

**Playbook**
- the governing doctrine that tells the system when and why to use those mechanisms

---

## 7. Sources of Playbook Formation

Playbooks should not be written from imagination.

They should compact from:

- empirical harness usage
- successful prompts and responses
- repeated failure patterns
- runtime edge cases
- neocorpus research
- canonical lessons

The compaction chain is:

`lineage -> repeated patterns -> playbook doctrine -> skills/hooks/operators`

This is the opposite of ad hoc prompting.

## 10. Standard Playbook Shape

Every playbook should eventually converge toward a standard shape:

- `PLAYBOOK.md`
- `PATTERNS.md`
- `ANTI-PATTERNS.md`
- `KAIZEN-LOG.md`
- `SKILLS/` or linked skill references when applicable
- `HOOKS/` or linked operator references when applicable

The exact filenames can evolve, but the roles should remain.

---

## 11. Edge Cases

## 11.1 A constitutional rule copied into a playbook

Allowed as a citation or derived operational rendering.
Not allowed as a replacement for the constitutional source.

## 11.2 A successful prompt that seems universally useful

Do not instantly treat it as playbook doctrine.
It becomes a playbook candidate only after repeated evidence or compaction review.

## 11.3 A surface that changes rapidly

Fast-changing surfaces still need playbooks.
The playbook should simply bias toward:

- invariant usage principles
- known volatility notes
- compact update cadence

## 11.4 A tool with no native memory

This strengthens the need for a playbook, not weakens it.
The playbook specifies how context is staged externally.

---

## 12. Alignment with the Harness Research

The harness/context-engineering research strongly supports this law:

- configuration hierarchies matter
- progressive disclosure matters
- file-backed context matters
- sub-agent/handoff patterns matter
- "prompting skill" is less scarce than specification skill

Playbooks are the place where specification skill becomes institutional memory for a harness.

They are how the system works **with** agents rather than against them.

---

## 13. Net Rule

If a recurring harness-specific lesson lives only in prompts, handoffs, or human memory, it is still un-institutionalized.

It becomes institutional only when it compacts into a playbook.
