## 4. Adjudicated Collisions

### 4.1 The new enforcement stream does not supersede the migration-control-plane stream

The 5.3 and CC92 artifacts could be misread as a separate initiative because they focus on renderer and validator hardening.

That reading is wrong.

They are a maturity-layer on top of the same migration program.
The tributary control plane supplies the first concrete subject matter that these enforcement primitives should eventually police.

### 4.2 The control plane is now ready for verification, not expansion

Wave 4 proved `executed`.
It did not yet prove `verified`.

That distinction matters because the control plane now has enough reality to be audited, but not yet enough proof to justify a broader intake or a wider Sigma tranche.

The next honest move is therefore:

- compute destination hashes
- advance the existing rows to `verified`
- harden the validator to understand the verified state

not:

- add more rows
- widen Sigma
- reopen disposition law

### 4.3 Communications naming drift is debt, not constitutional failure

The naming report contains real warnings, but almost all of them are best understood as:

- legacy artifact debt
- metadata under-annotation
- transitional packet formats

That debt should be classified before it is renamed.

### 4.4 The transitional root wrapper is still a real soft spot

The constitution report correctly keeps the root wrapper visible.

But it is not yet safe to remove it blindly because local Hazel rules or other edge integrations may still depend on its root path.

This makes the right immediate task:

- internalization readiness and bounded cutover planning

not:

- root deletion by aesthetic preference

## 5. Holistic Evaluation

This program has crossed an important line.

Earlier waves asked:

- what should survive?
- where should it land?
- how should the shell be stratified?

The current frontier asks:

- can the existing control plane survive verification?
- can the shell’s own law be rendered and checked deterministically?
- can drift be surfaced before it silently becomes doctrine?

That is a higher-quality problem.

The migration initiative is now less about deciding what the system is and more about proving that the system can govern what it already claims to be.

The main current hazard is no longer incoherence.
It is a three-part maturity trap:

- false completion
- premature strictness
- expansion before the first verified proof set stabilizes

The right posture is:

- verify first
- harden validators second
- classify debt third
- expand only after proof survives contact with enforcement

## 6. Wave 5 Boundary

Wave 5 should be a verification and enforcement-hardening wave.

It should do four things:

1. advance the existing 10 live rows from `executed` to `verified` with real `dest_artifact_hash` values
2. harden and wire the tributary validator so verified-state obligations and transition legality are checked under report-first discipline
3. classify the current communications naming warnings into clear remediation buckets without reopening all historical artifacts
4. produce an explicit internalization-readiness plan for the remaining transitional root operator

It should not do these things yet:

- add new tributary candidates
- widen Sigma beyond tranche 01
- bulk-rename communications history
- hard-delete the transitional root wrapper without a cutover plan
