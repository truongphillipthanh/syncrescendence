## 4. Ledger Doctrine

The imported artifacts imply a sharper ledger law than the shell currently states explicitly.

### 4.1 Live ledgers are not static documents

They are append-only or mutation-traceable intelligence surfaces whose value comes from:

- structured entries
- provenance
- freshness visibility
- cross-reference capacity
- analysis-to-dashboard compaction

### 4.2 Repo and exocortex must split the burden

The repo should hold:

- ledger constitutions
- schemas
- validators
- compacted snapshots
- ratified synthesis

The exocortex or other structured operational surface should hold:

- high-churn live entries
- query views
- dashboard projections
- working substrate state

## 5. Progressive-Disclosure Rule

The imported documents correctly insist that agents and humans should not consume the same fidelity surface by default.

The shell should therefore treat future live ledgers as three-layer systems whenever the domain justifies it:

1. substrate layer for structured entries and high-fidelity agent consumption
2. synthesis layer for analysis, compaction, and pattern extraction
3. dashboard or executive layer for human-resolution status and decision support

The repo normally ratifies layers `2` and `3` plus the contracts governing layer `1`.

## 6. Seven-Ledger Ambition, Current-Shell Discipline

The imported taxonomy proposes a seven-ledger system plus specialized domain ledgers.
That is directionally right, but the shell should proceed in this order:

1. ratify the generic ledger contract family
2. prove one or two domain ledgers end to end
3. add dashboard and decay rules only after the substrate behaves lawfully
4. expand the family only after the first ledgers compact real intelligence rather than ceremonial metadata

This keeps the shell from repeating the old pathology where the conceptual superstructure outran operational proof.
