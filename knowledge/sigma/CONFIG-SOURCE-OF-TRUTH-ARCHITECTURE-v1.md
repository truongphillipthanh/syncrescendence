# Config Source-Of-Truth Architecture v1

**Status**: live-v1
**Class**: sigma doctrine
**Purpose**: define the lawful source stack for shell configuration so behavior can project across surfaces without drift, duplicated doctrine, or hidden sovereigns

## 1. Decision

The shell's lawful configuration path is:

`canon -> Sigma -> constitutional config kernel -> derivative config surfaces -> rendered vendor artifacts -> config ledgers and reports`

This is the governing answer to the current configuration problem.

- canon and Sigma remain the doctrine-bearing source layers
- the constitutional config kernel becomes the direct source for lower config derivation
- playbooks, skills, hooks, schemas, templates, and ledgers are derivative surfaces with bounded burdens
- imported custom instructions remain high-resolution witnesses, not the final direct render source

If the shell renders lower configuration directly from vendor custom-instruction text, it imports vendor phrasing drift into every surface.
If it renders from a repo-native config kernel, each lower surface can stay native to its own grain without inventing a second personality.

## 2. Lawful Source Stack

Only the first three layers below may define shell-wide behavioral doctrine.
Everything after that is derivative, local, rendered, or append-only.

### 2.1 Canon

Canon holds the shell's macro law:

- strategic telos
- sovereignty boundaries
- office and federal burden boundaries
- ratification and compaction law
- trust-boundary law
- shell-universal anti-drift rules

Canon does not hold provider syntax, field-packing tactics, or tool-local workflow recipes.

### 2.2 Sigma

Sigma holds reusable secondary doctrine that translates macro law into repeated-use config judgment:

- cross-surface behavioral invariants
- imported custom-instruction interpretation doctrine
- projection and render law
- anti-pattern doctrine
- config architecture doctrine

Sigma interprets witnesses and defines stable secondary law.
It does not allow imported surface text to quietly become sovereign.

### 2.3 Constitutional Config Kernel

The constitutional config kernel is the shell's direct source-of-truth for lower-level configuration.
It is the typed middle layer between doctrine and projections.

It should carry:

- behavior atoms
- anti-pattern suppressions
- surface profiles
- projection priorities
- explicit omission rules
- source bindings
- renderability constraints
- source-versus-generated classification

This is the layer that says:

- which behaviors are shell-universal
- which surfaces must carry which subset
- what compression is lawful
- which details are vendor-specific packaging
- which artifacts are hand-authored, rendered, or append-only

### 2.4 Derivative Config Surfaces

Derivative config surfaces consume the config kernel and express it at native grain.
They may specialize locally inside their lawful scope, but they may not author a parallel shell personality.

Primary derivative families:

- playbooks
- skills
- hooks
- schemas
- templates
- rendered constitutions and vendor artifacts
- ledgers and reports

### 2.5 Rendered Vendor Artifacts

Rendered vendor artifacts are surface-native packaging such as:

- chat-app custom-instruction fields
- harness-specific constitution veneers
- tool settings stanzas
- agent profile blocks

They matter operationally, but they are not sovereign source files.
They are projections of upstream doctrine through the config kernel into a specific constraint surface.

### 2.6 Ledgers And Reports

Ledgers and reports preserve append-only or mutation-traceable evidence about render, validation, drift, and promotion.
They preserve state and proof.
They do not redefine doctrine.

## 3. Witness Rule For Imported Custom Instructions

The imported chat-app custom-instruction family is the highest-resolution behavioral witness currently available.
It is not the shell's final direct render source for lower-level configuration.

The lawful handling rule is:

1. preserve exact imported texts as source-proximate witnesses
2. interpret their stable payload in Sigma
3. distill that payload into the constitutional config kernel
4. project the kernel into lower config surfaces according to native grain
5. use the imported texts later as parity-check witnesses when vendor surfaces change

Therefore:

- exact imported custom-instruction texts belong in Sigma references as provenance-bearing witnesses
- Sigma doctrine explains what those witnesses mean
- the config kernel is what lower renderers should read directly
- lower surfaces should not paste, paraphrase, or diff vendor wording as if that wording were shell law

The governing sentence is:

`imported custom instructions are high-resolution witnesses, not the final direct render source`

## 4. Classification Of Surface Burdens

| Surface | What belongs there | What must not live there | Authoring mode |
| --- | --- | --- | --- |
| `canon` | macro telos, sovereignty, office law, ratification law, trust boundaries, shell-universal anti-drift law | vendor syntax, field budgets, tool-local tactics, runtime state | hand-authored only |
| `Sigma` | cross-surface config doctrine, witness interpretation, projection law, anti-pattern doctrine, secondary behavioral law | raw imported text pretending to be sovereign, office-local scratch, runtime receipts | hand-authored only |
| `config kernel` | behavior atoms, anti-pattern suppressions, surface profiles, projection priorities, omission rules, source bindings, renderability constraints | long workflow essays, provider-only packaging lore, append-only state | hand-authored source plus typed structured companions |
| `playbooks` | office- or surface-specific workflow doctrine, escalation logic, patterns, anti-patterns, examples | shell-wide identity law, duplicated macro doctrine, hidden policy | hand-authored, derivative |
| `skills` | bounded task protocols, checklists, inputs, outputs, evidence requirements, verification steps | macro personality, broad duplicate doctrine, silent permissions | hand-authored body with derivative packaging |
| `hooks` | executable policy, gates, deny or ask logic, audit emission, lawful transformation steps | prose personality, constitutional essays, hidden doctrine | code and config derived from policy |
| `schemas` | typed fields, enums, versions, required pointers, validation contracts, capability flags | motivational prose, shell personality, unnecessary vendor lore | hand-authored typed source |
| `templates` | placeholders, required headings, metadata slots, expected response shape, handoff molds | hidden doctrine, ad hoc policy, unique task decisions | hand-authored or rendered scaffold |
| `ledgers` | append-only render receipts, validation findings, drift detections, freshness state, promotion receipts, ratification joins | mutable doctrine, narrative law, stealth overrides | machine-appended or tightly controlled |

## 5. Projection Law

Lower surfaces should be derived by semantic distillation, not literal carryover.

The governing projection rules are:

1. preserve shell-universal behavior atoms before explanatory prose
2. preserve anti-pattern suppressions before examples or stylistic flourishes
3. if a lower surface has a native enforcement mechanism, encode the rule structurally instead of repeating it as philosophy
4. if a surface cannot express an atom, record explicit omission rather than allowing silent loss
5. keep vendor-specific packaging in surface profiles, renderers, or surface-local playbooks rather than in canon
6. require render and drift evidence to flow into ledgers or reports rather than living only in threads or memory

This means:

- playbooks explain and specialize
- skills operationalize bounded protocols
- hooks enforce
- schemas type and validate
- templates scaffold output shape
- ledgers record proof and mutation history

No lower layer is allowed to invent a new shell temperament because it lacked a clean source binding.

## 6. Shell-Universal Versus Vendor-Specific Content

The shell must keep universal doctrine separate from packaging that belongs only to one host surface.

### 6.1 Shell-Universal

These belong upstream in canon, Sigma, or the config kernel:

- epistemic discipline
- response-scaling law
- anti-pattern suppression
- provenance and receipt discipline
- office and sovereignty boundaries
- source-versus-generated law
- projection priorities
- omission and drift rules

### 6.2 Vendor-Specific Or Harness-Specific

These belong only in rendered surfaces, surface profiles, or local playbooks:

- field splits and UI budgets
- app-specific settings syntax
- agent-global partitioning rules
- tool-native permission grammar
- command wiring and slash-command packaging
- host-surface naming and placement details

Vendor packaging can constrain the render.
It cannot redefine the shell-universal atoms being rendered.

## 7. Anti-Drift Governance

The shell should govern configuration drift with the following standing rules:

- no derivative surface may author a second macro personality
- every rendered or maintained config surface should trace upward to the config kernel or an explicitly bounded local playbook
- every local specialization should remain inside office, harness, or task scope
- every render or validation event that matters should be receipted in a ledger or report
- when a witness family changes, update Sigma and the config kernel first, then rerender downstream artifacts second

The shell's behavior should therefore stabilize by source binding and typed projection, not by repeated manual restatement.

## 8. Immediate Operational Effect

This architecture establishes the constitutional config kernel as the shell's direct configuration source-of-truth beneath canon and Sigma.

Operationally, that means:

- canon remains the macro constitutional source
- Sigma remains the secondary doctrine and witness-interpretation layer
- the config kernel becomes the direct input to future lower-surface derivation
- playbooks, skills, hooks, schemas, templates, and ledgers remain lawful derivative families with explicit burdens
- imported custom instructions remain preserved as high-resolution witnesses for distillation and parity review, not as the text to render downward verbatim
