## Minimal Orientation To The System

### What Already Exists

The project already has meaningful implementation in:

- rendered config architecture from shared source files
- live agents across multiple machines
- runtime snapshots and reconciliation artifacts
- structured memory/event flow
- exocortex taxonomy and account/identity mapping
- a boundary contract for repo vs dotfiles vs runtime vs ontology
- a first successful CLI<->web relay using Cowork + Claude in Chrome

So this is not a "what are AI agents?" prompt.
It is a governance and systems-architecture verification prompt around an already partially working stack.

### What Broke Historically

The repo accumulated:

- duplicate folders
- stale paths
- orphaned prompts and responses
- mixed logs, reports, scaffolds, and doctrine
- partial migrations that were never fully enforced

### What CRUSH / Nucleosynthesis Means

To remediate that, a major flattening/reclassification effort was performed:

- non-canonical material was flattened into a large corpus
- filename and historical directory structure ceased to be trusted as semantic truth
- material was reclassified by semantic topic
- a **neocorpus** was created as a distilled doctrine/compendium layer

This made the repo more traversable and semantically coherent, but it did not by itself solve:

- where constitutional vocabulary lives
- where executive intentions live
- how backlog/program should relate upward to those layers
- how prompts, responses, logs, handoffs, outputs, and playbooks should be separated and enforced

## Mission

We are not asking you to design our architecture from scratch.

We are asking you to verify whether our current remediation thesis matches current reality:

1. the harness is the product
2. repo-as-coordination-surface is the right async authority pattern
3. one constitutional source + many harness-specific playbooks is better than one flattened universal prompt
4. prompts/responses/logs/handoffs/outputs should be treated as different artifact classes
5. a federal architecture is the right metaphor for keeping constitutional law, executive intention, backlog/program, harness-local doctrine, and ontology registry distinct

We want verification, precedent, and recent capability changes — especially anything from late 2025 through March 2, 2026 that could materially change how we should architect this.

## Current Thesis To Stress-Test

The current internal diagnosis is:

- CRUSH / nucleosynthesis solved a traversability problem by flattening and reclassifying a large, crufted repo
- neocorpus now gives a much more distilled doctrine layer
- the live multiagent->ontology pipeline is ahead of the repo's filing/communications constitution
- Rosetta Stone and Intent Compass are still conceptually central, but their live authoritative file paths appear unresolved
- backlog exists and is active, but is not tightly linked upward to constitutional vocabulary and executive intention
- harness-specific playbooks are under-formalized
- the communications layer is under-defined: prompts, responses, logs, handoffs, and outputs are still too blended

We need to know whether the remediation direction below is actually sound:

- **constitution**: shared law, vocabulary, boundary contracts
- **executive**: active intentions / priority compass
- **program**: backlog and implementation map
- **playbooks**: per-harness doctrine
- **communications**: prompt/response/handoff/log/output law
- **registry**: ontology projection

This should be treated as a governance and architecture question, not a cosmetic folder-organization question.

## What We Need Verified

### 1. Harness-Specific Playbooks

Verify current precedent on:

- whether strong AI teams are converging on per-harness / per-agent playbooks rather than one universal mega-prompt
- whether progressive disclosure, modular rules, skills, or layered config systems are the dominant pattern
- whether there are current examples of "shared constitution + surface-specific doctrine"

Focus on:

- Claude Code / CLAUDE.md patterns
- Codex / GPT coding harness patterns
- Gemini / Google coding harness patterns
- MCP / rules / skills / profile systems

### 2. Repo-As-Authority

Verify whether current leading teams and serious practitioners are converging on:

- filesystem/repo as durable coordination surface
- dispatch-as-file or artifact-based handoff patterns
- git as the durable state machine / publication event

We do not need generic "version control is useful" commentary.
We need current patterns close to async agent coordination, repo sovereignty, or local-first AI operations.

### 3. Artifact-Class Separation

Verify whether there is precedent or strong current practice for separating:

- prompts
- responses
- logs
- ADRs / decisions
- outputs / reports / generated artifacts
- playbooks / operating doctrine

Especially relevant:

- knowledge repos
- AI engineering repos
- serious Obsidian-as-code or docs-as-ops practices
- incident/runbook/retrospective patterns from adjacent engineering fields

### 4. Federal Layering

Verify whether the following architectural split is sound or if there is a better established equivalent:

- constitutional layer
- executive priorities / active intention layer
- administrative backlog / program layer
- local playbook layer
- registry/projection layer

You do not need to find someone using the word "federalism."
You do need to determine whether this layered governance model is structurally supported by current best practice.

In concrete terms, verify whether modern AI/harness-heavy repos benefit from clearly separated layers for:

- durable constitutional law
- active priority / agenda
- administrative program / backlog
- harness-specific operating doctrine
- typed downstream registry / projection

### 5. Recent Product / Capability Changes

Verify any late-2025 to Mar 2, 2026 product changes that matter here, including:

- Claude Code / Claude Desktop / Cowork / Claude in Chrome
- OpenAI Codex / ChatGPT coding harness changes
- Gemini CLI / Google AI Studio / NotebookLM changes
- Obsidian CLI / Headless / plugin ecosystem changes

We are specifically looking for anything that changes:

- local-first file-grounded workflows
- harness modularity
- browser + local-folder workflows
- better ways to structure playbooks or working memory
