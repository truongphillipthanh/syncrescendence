# FAILURE-MODES-AND-RECOVERY

This document outlines the most likely failure modes for each migration wave and the deterministic actions required to recover. The primary recovery tool is the break-glass account (`truongphillipthanh@icloud.com`), whose access is verified in Wave 0.

| Wave | Failure Mode | Recovery Action |
|---|---|---|
| **Wave 1** | Invitation to `syncrescendence@gmail.com` is not received or fails. | 1. Check spam/junk folder in the `syncrescendence@gmail.com` inbox. <br> 2. As the current owner, revoke the pending invitation on the platform. <br> 3. Re-send the invitation, double-checking the email address. |
| **Wave 2** | **Critical:** GitHub organization ownership transfer fails mid-process, or the new owner cannot log in after transfer. | 1. **Immediate Action:** The legacy owner (`truongphillipthanh@icloud.com`) must immediately attempt to perform an administrative action. <br> 2. If the legacy owner still has owner rights, cancel the transfer if the option is available. If not, immediately re-invite the legacy owner as an owner from the new owner account (if accessible). <br> 3. **If `syncrescendence@gmail.com` is owner but locked out:** Use the break-glass account (`truongphillipthanh@icloud.com`) to contact GitHub support, providing evidence of ownership. This is the last resort. |
| **Wave 2** | OAuth client or service account ownership transfer fails on Google Cloud. | The operation is usually not atomic. Revert ownership on the specific credential that failed back to the original account. The system remains functional, and the transfer can be re-attempted. |
| **Wave 3** | `MANUS-RUN-BATCH` for Git identity cutover fails. | The script is idempotent. The operator can safely re-run the batch. If it continues to fail, the operator can manually execute the `git config` commands from the script on their local machine. The risk is low as it only affects the local developer environment. |
| **Wave 3** | New commits are not being signed correctly. | 1. Verify the correct GPG/SSH key ID was provided to the script. <br> 2. Ensure the GPG/SSH agent is running correctly on the local machine. <br> 3. Confirm the public key has been uploaded to the `syncrescendence@gmail.com` GitHub account. |
| **Wave 4** | Final audit (`MANUS-RUN-BATCH`) shows unexpected members remain in the organization. | 1. Log in as the owner (`syncrescendence@gmail.com`). <br> 2. Manually remove the unexpected members from the organization via the GitHub UI. <br> 3. Re-run the audit batch to confirm resolution. |
| **All Waves** | **Global Lockout:** All active owner accounts (`syncrescendence@gmail.com` and legacy) lose access simultaneously. | This is the scenario the break-glass account is for. <br> 1. Use the `truongphillipthanh@icloud.com` credentials to log into the platform (e.g., GitHub). <br> 2. As this account was a previous owner, it has standing to contact support and prove identity to recover the organization. <br> 3. Once access is restored for one account, use it to re-invite and restore the others. |
