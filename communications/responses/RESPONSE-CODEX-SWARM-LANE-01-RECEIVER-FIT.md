## Missing Or Under-Specified Receiving Surfaces

1. A live disposition registry is missing. Tranche T defines disposition classes, but the shell does not yet expose one canonical receiver-side registry that records per-artifact destination, rationale, and state.
2. `pedigree/archive-manifests/` is too thin. Archive-manifest-only is named as a lawful disposition, but the lane does not yet look like an active intake surface.
3. `knowledge/canon/` is underdeveloped. The law distinguishes canon from references, but the actual lane is still mostly conceptual.
4. Executive sublanes are lightly instantiated. `briefings/`, `escalations/`, and `summits/` have templates, but not enough live exemplars to prove migration fit for `-SOVEREIGN` density.
5. There is no explicit receiver law for source shedding manifests. The shedding rule exists conceptually, but not yet as a compact operational contract for where externalized or culled source families are tracked.
6. The ontology receiving boundary is underspecified. The shell says ontology is projection, but it is still too easy for migration work to confuse typed projection with durable semantic authority.
7. Communications law names `dispatches`, but the physical lane does not appear as a dedicated first-class sublane under `communications/`.
8. Runtime versus office-local evidence boundaries remain fragile. Both are governed, but tributary migration could still misfile platform logs or operational state between `offices/`, `runtime/`, and `communications/logs/`.
9. Program-scale migration accounting is not yet registry-backed. `program/` holds the tranche, but not a migration ledger that can absorb thousands of receiver decisions without backlog sprawl.
10. Cross-lane promotion criteria are still scattered. The shell has good law, but a receiver operator or contract that says "if class X with burden Y, land in lane Z" is still distributed across multiple documents.

## Top 10 Receiver-Side Gaps That Could Cause Migration Failure Or Relapse

1. No canonical disposition registry. Without one receiver-side ledger, migration will relapse into narrative decisions spread across backlog notes, receipts, and local memory.
2. Canon versus reference remains too soft. If `knowledge/canon/` stays thin, doctrinal material from pre-schematic tributaries will either bloat `references/` or leak upward into law.
3. Archive-manifest-only is not yet operationally real. Without a strong manifest lane, the shell will either hoard low-yield source matter or cull it without durable traceability.
4. Executive receiving surfaces are structurally present but semantically underloaded. This risks either overfilling `executive/` with history or under-preserving real sovereign meaning.
5. Ontology boundary discipline is not yet compact enough. Migration teams could let registry projection quietly become a second canon or a hidden control layer.
6. Source shedding lacks a hardened receiver contract. The shell knows it should stop warehousing raw source matter, but the actual landing pattern for pointer plus receipt plus synopsis is not yet standardized.
7. Dispatch metadata is still thinner than communications law implies. If dispatch records remain implicit, future lineage audits will degrade as communication volume rises.
8. Office, runtime, and communications logs can still be confused under pressure. High-volume migration will magnify any ambiguity between local work logs, runtime evidence, and federal communications history.
9. Program lane is not yet shaped for migration census at shell scale. A tranche document is not enough to manage broad tributary intake without a companion registry or scoreboard.
10. Receiver law is distributed across too many documents. The shell has strong parts, but migration operators still have to synthesize from many surfaces instead of consulting one compact receiver-fit contract.

## Net Recommendation

The successor shell is already sufficient to begin tributary migration by class. It is not sufficient for indiscriminate broad-volume migration.

Recommended order from the receiver side:

1. continue immediate migration into `communications/`, `pedigree/`, `offices/`, `playbooks/`, `validated-patterns/`, `operators/`, and `knowledge/references/`
2. strengthen `knowledge/canon/`, `executive/`, and `pedigree/archive-manifests/` before sending large doctrinal, sovereign, or shedding-heavy classes
3. create one canonical disposition registry before any high-volume tranche
4. formalize the receiver contract for source shedding and ontology/exocortex projection so repo sovereignty stays intact under scale

The shell has the right architecture. The remaining work is to harden the receiver interfaces that sit between lane law and migration volume.
