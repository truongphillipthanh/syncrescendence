## Returned Content

The correct future-facing design thesis is:

- the exocortex should unify cognitive organs operationally by giving them a governed control plane across external surfaces
- the ontology should unify them legibly by projecting typed, queryable views of repo-ratified state and exocortex status
- neither layer may become a second canon, a shadow executive, or an unreviewable semantic court

In this architecture, the repo remains sovereign because it is still the only place where durable law, doctrine, and direction-changing artifacts are ratified. The exocortex is the shell's distributed action surface. The ontology is the shell's typed visibility surface. Together they should reduce fragmentation across offices, operators, runtime, communications, and external tools, but only by making those relations inspectable and governable, not by absorbing constitutional authority.

## Design Thesis

Exocortex plus ontology should function as a unifier of cognitive organs in two different but complementary ways:

1. The exocortex unifies by coordination.
   It gives offices, programs, operators, and channels a common external execution fabric, common connector identity, common receipts, and common control-plane visibility.
2. The ontology unifies by projection.
   It gives the shell one typed graph of entities, relations, registries, statuses, and lineage snapshots so different organs can query the same projected structure without inventing local taxonomies.
3. The repo unifies by authority.
   It remains the ratifying center for law, doctrine, lane contracts, migration decisions, and all direction-changing semantic commitments.

The key design rule is therefore:

**exocortex coordinates, ontology projects, repo ratifies.**

That prevents a hidden authority layer while still allowing large-scale coherence across increasingly externalized operational surface area.

## Top Risks

### If ontology overreaches

1. Projection starts being treated as meaning.
   Teams begin reading typed entities and relations as the semantic authority rather than as a view over repo-defined truth.
2. Query convenience outruns doctrinal discipline.
   Whatever is easy to ask in ontology starts displacing what is lawfully stated in repo documents.
3. Registry schemas silently harden into doctrine.
   Field names, entity classes, and relation types become de facto canon even when no constitutional or doctrinal artifact ratified them.
4. Migration work gets distorted toward what fits the graph.
   Tributary material gets compacted to satisfy current ontology shapes instead of current shell law and future utility.

### If exocortex becomes hidden authority

1. Connector state starts deciding reality.
   A live integration graph or SaaS topology begins to determine semantic priority because it is more current than the repo.
2. Runtime surfaces become policy by momentum.
   What is possible in Slack, Notion, ClickUp, Linear, or other surfaces starts steering the shell more than explicit executive and program artifacts.
3. Verification state is mistaken for legitimacy.
   A verified connector may be safe and real, but it is not therefore semantically authoritative.
4. Operational memory exits reviewable lanes.
   High-value knowledge accumulates in tools, chats, automations, and dashboards without being projected back as receipts, statuses, or repo-ratified artifacts.

## What Belongs Where

### Repo-resident

- constitutional law
- office burdens and role definitions
- executive direction and program decisions
- communications artifacts that carry instruction, response, assessment, or ratified handoff
- playbooks, validated patterns, operator code, migration law, and disposition contracts
- durable doctrine about how surfaces may be used
- any direction-changing artifact

### Registry-projected into ontology

- typed surface inventory
- teleology mappings
- connector manifests
- verification and readiness status
- auth-dependency edges
- lawful dispositions, provenance manifests, and migration ledgers
- normalized references to repo-ratified artifacts

These are projections because they should be queryable and composable, but they must remain downstream of repo authority.

### Exocortex-native

- live connector sessions and topology
- execution receipts generated at the edge of external tools
- bounded operational state for automation, routing, channeling, and synthesis
- transient surface-local scaffolds that do not deserve durable repo residence

Exocortex-native does not mean semantically authoritative. It means operationally real but constitutionally subordinate.

## Tributary Migration Implications Now

Future unification should already shape migration practice:

1. Migrate tributaries by authority class, not by folder recovery.
   If something is law, doctrine, playbook, pattern, pedigree, or operator logic, land it in the repo. If it is only inventory, status, disposition, or topology, prefer registry form. If it is raw operational burden better carried off-repo, externalize it with a pointer and receipt.
2. Prefer projection-ready compaction over narrative piles.
   Tributary promotions should produce artifacts that can later be expressed as registries, manifests, or typed status surfaces without reinterpreting their meaning.
3. Treat source shedding as a control discipline, not a storage trick.
   The point is to keep repo sovereignty compact while preserving provenance and queryability.
4. Do not migrate semantic authority into connector graphs.
   As the exocortex holds more operational surface than the repo, migration must become stricter about what remains ratified in-repo.

## Next Concrete Design Deltas

1. Create an explicit non-duplication contract for ontology schema changes.
   Any new ontology entity, relation, or field family that could alter meaning should require a corresponding repo-ratified artifact or amendment.
2. Add a formal semantic-authority split for control-plane artifacts.
   Surface registry, teleology, connector manifest, and status should each declare that they describe execution topology or projection state, not doctrine.
3. Promote a single migration disposition registry.
   Tributary decisions should be recorded in a typed ledger that cleanly separates `repo_resident`, `registry_projected`, `exocortex_native`, `pedigree_retained`, and `archive_manifest_only`.
4. Make projection lineage explicit in ontology.
   Every projected entity should carry source artifact pointers so query users can trace back to the ratifying repo surface.
5. Harden the rule that direction-changing outputs must land in repo artifacts before they are treated as institutional truth.
   This keeps live tool activity from silently becoming executive or doctrinal authority.
6. Add review gates for high-fanout surfaces.
   Slack, Notion, ClickUp, Linear, Atlassian, and similar hubs need stronger receipt, verification, and extraction discipline because they are the most likely hidden-authority surfaces.
7. Separate teleology from command.
   Teleology should explain what a surface is for; it must not assign policy or make prioritization decisions that belong to executive and program layers.

## Bottom Line

The exocortex and ontology should become the shell's unifying cognitive fabric without becoming its second scripture.

- the exocortex should hold increasing operational surface area
- the ontology should hold increasing typed visibility
- the repo should remain the only ratifying seat of law, doctrine, and direction

If that split is maintained, the shell can scale across many cognitive organs and many external surfaces without creating a hidden sovereign above the constitution.
