# Phase 0 & 1: Risk Register

This document outlines the top failure modes and mitigation strategies for the initial phases of the CC81 identity cutover.

| Risk ID | Description | Impact | Mitigation | Owner |
|---|---|---|---|---|
| **R-01** | **API Rate Limiting** | Incomplete or failed snapshot capture due to exceeding API call limits imposed by the platforms. This would result in an incomplete evidence baseline. | - Implement exponential backoff and retry logic in all snapshot scripts.<br>- Stagger the execution of snapshot commands across different platforms to distribute the load.<br>- Where possible, use bulk data export APIs instead of iterative queries.<br>- Contact platform support in advance for temporary rate limit increases if necessary for large-scale snapshots. | commander |
| **R-02** | **Insufficient Permissions** | Snapshot scripts fail because the associated service accounts or API tokens lack the necessary read-only permissions for all required resources. | - Conduct a thorough permission audit (SEC-01 gate) before Phase 1.<br>- Use a dedicated, role-based service account for each platform with explicitly defined read-only access.<br>- Test each snapshot command against a non-production or test resource on each platform to validate permissions (TECH-03 gate). | psyche |
| **R-03** | **Incomplete Snapshot Schema** | The `PHASE1-SNAPSHOT-SCHEMA` fails to capture a critical configuration item or user attribute, leading to a blind spot in the pre-cutover state. | - The schema will be reviewed and approved by representatives from the security, identity, and platform teams.<br>- The schema will be version-controlled in Git.<br>- A dry-run of the snapshot process will be performed, and the output will be manually reviewed against the platform's UI to ensure completeness. | human |
| **R-04** | **Evidence Tampering or Loss** | Snapshot evidence is altered, lost, or inaccessible after being collected, invalidating the integrity of the pre-cutover baseline. | - All evidence will be stored in an immutable, versioned object store (e.g., S3 with object locking).<br>- Each piece of evidence will have a corresponding `EVIDENCE-RECEIPT-TEMPLATE` containing a cryptographic hash (SHA256) of the artifact.<br>- Access to the evidence store will be restricted to a minimal set of authorized personnel and service accounts. | psyche |
| **R-05** | **Tooling or Script Failure** | The `commander` or `manus` tooling fails during execution due to unhandled errors, environmental issues, or bugs in the snapshot scripts. | - All snapshot scripts will include robust error handling and logging.<br>- The execution environment will be standardized and containerized to ensure consistency.<br>- All scripts will be peer-reviewed and tested in a development environment before being used in production. | commander |
| **R-06** | **Platform API Changes** | A platform provider makes a breaking change to their API between the time of testing and the execution of the cutover, causing snapshot scripts to fail. | - Pin all API client libraries to specific versions.<br>- Subscribe to developer change logs and mailing lists for all 14 platforms.<br>- Re-run connectivity tests (TECH-02 gate) immediately before initiating Phase 1. | manus |
| **R-07** | **Human Error** | An operator makes a mistake during a manual step, such as providing the wrong parameter to a script or misinterpreting a result. | - Automate as much of the process as possible, minimizing manual intervention.<br>- All manual steps will be clearly documented in the runbook with explicit instructions and expected outcomes.<br>- A second operator will review all manual actions and command inputs before execution (four-eyes principle). | human |
