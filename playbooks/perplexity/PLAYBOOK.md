## 0. What This Surface Is For

Perplexity (Augur role) is the external verification and current-reality surface.

Its proper role is:
- verify claims
- check current capability reality
- find disconfirming evidence or exceptions
- provide citation-backed support for or against a thesis

Perplexity is not the main architectural synthesizer and not the constitutional authority.

## 1. Native Grain

Perplexity’s native grain is:
- fast verification
- citation-rich response structure
- product/release/current-reality lookup
- narrower question answering with source support

It performs best when:
- the verification target is concrete
- acceptable source classes are stated
- current reality matters
- the packet asks for explicit source discipline

## 2. What Perplexity Is Bad At

Perplexity is risky when:
- asked to do deep architecture from scratch
- trusted blindly on secondary/community sources
- treated as if citation count equals correctness
- allowed to drift into a broad design role better handled by Oracle

## 3. Proper Role in Triangulation

Perplexity should answer:
- is this currently true?
- is there official or primary-source support?
- what current product or ecosystem reality bears on this thesis?

It should usually follow Oracle, not replace Oracle.

## 4. Packet Doctrine

A good Perplexity packet should specify:
- the exact claim or distinction to verify
- why the verification matters
- preferred source classes
- what counts as unacceptable evidence
- the return artifact and bridge path

This is not “ask the web.” It is verification under explicit evidentiary constraints.

## 5. Output Doctrine

Treat Perplexity returns as:
- verification inputs
- current capability reality checks
- citation-bearing challenge material

Do not treat them as:
- constitutional ratification
- substitute for repo assessment
- universal proof merely because citations exist

## 6. Common Use Cases

Perplexity is the right surface for:
- current product or feature verification
- release/update checks
- API/docs/status triangulation
- checking whether a proposed architecture matches current ecosystem reality

It is the wrong surface for:
- designing the shell from first principles
- replacing the repo’s own assessment layer
- unstructured broad sensemaking

## 7. Net Rule

Use Perplexity as the verifier and current-reality check.
Constrain the evidence class.
Land the result.
Assess it before adopting it.
