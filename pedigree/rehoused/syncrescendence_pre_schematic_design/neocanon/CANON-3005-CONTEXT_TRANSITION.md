# CANON-3005: CONTEXT TRANSITION PROTOCOL
## Cognitive Continuity Across Sessions

> You're not compressing a library. You're designing the librarian.

AI sessions end. Context windows fill. Platforms change. The work continues. The Context Transition Protocol is the unified interface for managing cognitive continuity across every kind of session boundary — end, start, mid-stream, cross-platform, cross-agent.

The practitioner does not choose the mode. The AI does. The practitioner invokes the protocol; the system analyzes context and produces the appropriate artifact. This is the terminal architecture thesis: AI mediates complexity. The practitioner maintains strategic sovereignty without operational micromanagement.

---

## PART I: THE UNIFIED INTERFACE

### One Interface, Multiple Modes

There is one operation: context transition. It has three modes that activate based on context analysis. The modes combine freely.

**How to invoke** (any of these triggers the protocol):
- "Handle context transition"
- "Close this session"
- "Continue from [previous work]"
- "Context is getting long — compress"
- Or implicitly: approaching context limits, natural conclusion reached, extended silence

**What happens**:
1. The AI analyzes current context state
2. Determines which modes apply
3. Generates a transition artifact with appropriate sections activated
4. Presents for practitioner review

The practitioner does not need to understand the mode selection logic. They need to know that invoking the protocol produces the right artifact for the situation.

### The Three Modes

**Culmination Mode**: Activated when a thread ends with value to preserve. Captures insights, identifies candidates for permanent knowledge, creates historical record. Output focus: what was learned? What should persist beyond this session?

**Continuation Mode**: Activated when work will resume across a session boundary. Enables seamless resumption without re-explanation. Output focus: what state must be restored? What is the entry point for the next session?

**Compression Mode**: Activated when context approaches capacity limits mid-stream. This is lossy compression in Shannon's precise sense -- preserving essential state while reducing token footprint, accepting irreversible information loss bounded by the protocol's fidelity guarantees. Output focus: what must survive? What can be regenerated from pointers?

These modes are not mutually exclusive. Most real transitions activate two or three simultaneously. A productive session ending with work to resume activates culmination plus continuation. Approaching context limits while work continues activates compression, possibly combined with continuation.

## PART II: CONTEXT ANALYSIS

### State Variables

The AI evaluates six factors to determine which modes activate:

| Variable | What It Assesses | How It Affects Mode Selection |
|----------|-----------------|-------------------------------|
| **Position** | Where in the thread — start, middle, end? | End → culmination. Middle → compression. Start → continuation (injection). |
| **Capacity** | Percentage of context window consumed | >70% → compression urgency increases |
| **Value Density** | Were significant insights generated? | High density → culmination warranted |
| **Continuation Need** | Will work resume? | Yes → continuation artifact needed |
| **Temporal Gap** | Time since last session | Large gap → deeper context restoration needed |
| **Complexity** | Simple handoff or rich state? | Rich → more comprehensive artifact |

### Mode Selection Logic

```
IF thread ending OR explicit close request:
    IF significant insights generated:
        INCLUDE culmination
    IF work will resume:
        INCLUDE continuation

IF context > 70% capacity AND thread continuing:
    INCLUDE compression

IF thread starting AND prior context exists:
    INCLUDE continuation (as injection)

GENERATE composite artifact from selected modes
```

This logic is a guide for the AI, not a rigid algorithm. The AI exercises judgment about what "significant insights" means, whether work will resume, and what level of compression is appropriate.

## PART III: THE TRANSITION ARTIFACT

### Unified Structure

One adaptive template. Sections activate based on mode selection. No empty sections. No redundant content.

```markdown
# CONTEXT TRANSITION ARTIFACT
**Generated**: [timestamp]
**Thread**: [reference/title]
**Modes Active**: [Culmination | Continuation | Compression | combination]

---

## CONTEXT STATE SUMMARY
[Always present. Brief description of cognitive state.]
Position: [where in the work]
Capacity: [context utilization if relevant]
Focus: [current objective]

---

## SYNTHESIS [activated by Culmination]

### Trajectory
[How the work evolved — key turns, pivots, discoveries]

### Key Insights
1. **[Title]**: [Elaboration]
   - Implication: [what this means for practice]

### Canonical Candidates
[Insights stable enough to graduate to permanent knowledge]

### Unresolved Questions
[What remains open]

---

## CONTINUATION STATE [activated by Continuation]

### Essential Context
[Minimum viable context for effective resumption]
- Key decisions made
- Active constraints
- Current objective

### Entry Point
"We were [state]. The next step is [action].
Key context: [critical facts]."

### Resource Pointers
[Relevant documents, prior artifacts, reference material]

---

## COMPRESSION [activated by Compression]

### Compressed State
[Single paragraph capturing essential current state]

### Preserved Elements
[Critical items that must survive compression]

### Regeneration Guide
[Pointers to expand detail if needed]

### Discarded Elements
[What was removed and why — enables audit]

---

## NEXT ACTIONS
[Always present if work continues]
1. [Immediate next step]
2. [Following step]
```

### Common Combinations

| Scenario | Modes | Dominant Sections |
|----------|-------|-------------------|
| Productive session ending, will resume | Culmination + Continuation | Full Synthesis + Entry Point |
| Context limit approaching mid-work | Compression | Compressed State + Regeneration Guide |
| Starting new session on prior work | Continuation (consumption) | Entry Point as injection |
| Thread naturally concluding, no resume | Culmination only | Full Synthesis + Canonical Candidates |
| Quick checkpoint, minimal value | Continuation only | Essential Context + Entry Point |

### Artifact Quality Criteria

A good transition artifact:
- **Stands alone**: Readable without access to the original thread
- **Mode-appropriate**: Includes only sections relevant to selected modes
- **Actionable**: Clear next steps if work continues
- **Auditable**: Shows what was preserved and what was discarded
- **Efficient**: Minimum tokens for maximum context restoration -- rate-distortion optimality in Shannon's framework

## PART IV: TRIGGER MANAGEMENT

### Automatic Triggers

The AI detects conditions and offers transition:
- Context utilization exceeds 80%
- Natural conclusion detected — goal achieved, question answered
- Extended gap in conversation — session likely ending
- Significant insight generated — worth capturing now

Automatic triggers are offers, not commands. The practitioner can decline, defer, or modify.

### Explicit Triggers

The practitioner requests directly:
- "Let's close this session"
- "Checkpoint here"
- "I need to switch platforms"
- "Prepare continuation context for later"

### Practitioner Override

The AI proposes; the practitioner disposes. Override options:
- "Just culmination — I won't continue this"
- "Skip synthesis — just give me continuation context"
- "Compress more aggressively"
- "Include [specific element] I need preserved"

The practitioner's override is final.

## PART V: SESSION LIFECYCLE

### The Full Cycle

```
SESSION START
    │
    ▼
INJECT CONTEXT ◄── Prior transition artifact (continuation mode)
    │
    ▼
WORK SESSION ◄── Context analysis (ongoing, background)
    │
    ▼ (if capacity > 70%)
COMPRESS IF NEEDED ── Mid-session transition (compression)
    │
    ▼
CLOSE SESSION ── End-session transition (culmination + continuation)
    │
    ▼
ARCHIVE ARTIFACT ── Store, version control
```

At session start: load the most recent relevant transition artifact as context injection. The artifact was designed for this purpose — it contains exactly what the next session needs to resume effectively.

During the session: context analysis runs passively. The AI monitors capacity utilization and value density without interrupting work.

At session end: the protocol generates the artifact that the next session will consume. The cycle is closed.

### Cross-Platform Transitions

When work moves between platforms or agents:

The source generates a transition artifact with continuation mode active, framed for the target's function. The practitioner transfers the artifact (via file system, clipboard, or shared storage). The target receives the artifact as context injection, calibrated to its specific cognitive mode.

**Handoff metadata** appended for cross-platform transitions:
- Source: [who/what generated this]
- Target: [who/what will consume it]
- Rationale: [why the work is moving]
- Task: [what the target should do]
- Translation notes: [reframing needed for target's mode]

## PART VI: THE DISTILLATION INTERFACE

### From Transition Artifact to Permanent Knowledge

Transition artifacts are fluid — they serve the moment. Some contain insights that deserve permanence. The distillation interface manages this graduation.

```
Transition Artifact (fluid)
    │
    ▼ [practitioner review]
Canonical candidate identified
    │
    ▼ [validation period]
Insight proves stable across 3+ sessions
    │
    ▼ [corpus update]
Permanent knowledge document created or updated
    │
    ▼ [propagation]
Updated knowledge available to all agents/sessions
```

### Graduation Criteria

An insight graduates from transition artifact to permanent knowledge when:
- It has proven stable across three or more sessions
- It represents framework evolution, not task-specific detail
- It fills an identified gap in permanent knowledge coverage
- It aligns with constitutional principles
- The practitioner approves graduation

### The Forgetting Interface

Not all transitions warrant preservation. The default is archive, but pruning is necessary:

- **Archive** (default): Kept for potential reference
- **Prune**: Transitions with no lasting value deleted
- **Merge**: Related transitions consolidated
- **Graduate**: Insights promoted to permanent knowledge

Monthly hygiene: review archived transitions, prune low-value, consolidate related. Transition artifacts that haven't been referenced in 90 days are candidates for pruning.

## PART VIII: MEASUREMENT

### Success Metrics

- **Time to productive state** after session start — shorter is better
- **Context restoration completeness** — measured by follow-up questions needed (fewer is better)
- **Compression ratio** versus semantic fidelity maintained
- **Canonical graduation rate** from transition artifacts — indicates insight quality
- **Practitioner satisfaction** — does the artifact actually help when resuming?

These metrics are tracked informally. The practitioner notices when transitions work and when they don't. Over time, the protocol improves through accumulated experience.
